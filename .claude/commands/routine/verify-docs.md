# /verify-docs

ドキュメント（docs/）の内容がコードの実装と一致しているかを検証するスラッシュコマンドです。

## 確認観点

各ドキュメントについて以下を確認します：

- コマンド・スクリプトが `package.json` と一致しているか
- ファイルパス・コンポーネント名が実在するか
- API・IPCチャンネル名が正しいか
- 機能説明が実装と一致しているか
- 廃止された機能・コードが残っていないか

## 実行手順

1. **対象ドキュメントの取得**
   - `docs/**/*.md` をGlobで取得
   - TEMPLATE.md, WRITING-GUIDE.md などのテンプレートは除外

2. **チェックリストファイルの作成/更新**
   - `doc-review-checklist.md` をプロジェクトルートに作成
   - 取得したファイルをカテゴリ別に整理してチェックボックスを設置

3. **カテゴリ順に検証**
   - TodoWriteで進捗を管理
   - 各ファイルを読み、関連コードと照合
   - 不整合を発見したらチェックリストに記録

4. **検証内容（カテゴリごと）**

   **setup/**:
   - `package.json` のスクリプトと照合
   - 環境変数・ポート番号の確認

   **features/**:
   - 該当サービス・ハンドラーのコードと照合
   - IPCチャンネル名の確認

   **architecture/**:
   - ディレクトリ構造・ファイル構成と照合
   - データ形式・API仕様の確認

   **screens/**:
   - コンポーネントファイルの存在確認
   - Props・機能説明の照合

   **testing/**:
   - テストコマンド・テスト名の確認
   - 手動テスト項目と機能の照合

5. **サマリー更新**
   - チェックリストの集計表を更新
   - 要修正リストを作成

## 出力形式

`doc-review-checklist.md` に以下を記録：

- 各ファイルの確認状況（チェックボックス）
- カテゴリごとの不整合メモ
- サマリー表（確認完了数、不整合数）
- 要修正リスト（ファイル、問題内容、優先度）

## 優先度の基準

- **高**: 誤ったコマンド、存在しないファイルパス、間違ったAPI名
- **中**: 古い情報、不完全な説明、軽微な不一致
- **低**: 表現の改善、細かな修正提案

## 実行タイミング

- リリース前の最終確認
- 大きな機能追加・変更後
- 定期メンテナンス（月1回程度）
